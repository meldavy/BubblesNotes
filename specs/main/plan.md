# Implementation Plan: Notes App with AI Integration

**Branch**: `bubbles-notes` | **Date**: 2026-03-18 | **Spec**: [spec.md](./spec.md)
**Input**: "Build a notes app like google keep"

## Summary

This feature implements a Google Keep-like notes application with the following core capabilities:
- Quick note creation via a persistent editor at the top of the dashboard
- Google OAuth authentication for user identity and data isolation
- Markdown note editing with real-time preview
- URL preview generation for shared links
- File attachments with encryption
- Tagging system with chip input interface
- Content search across notes, tags, and attachments
- AI-powered note enhancement (summary, title, tags) via OpenAI-compatible API
- Infinite scroll with lazy loading for the notes grid
- REST APIs for programmatic access

**Technical Approach**: Ktor backend with PostgreSQL/SQLite database, server-side React frontend bundled with the server, Google OAuth authentication, async job processing for AI enhancement, and file encryption using centralized server key.

## Technical Context

**Language/Version**: Kotlin 2.3, JavaScript (ES2021+)  
**Primary Dependencies**: Ktor 3.4.1, Jackson, kotlinx.serialization, React 18+, Marked.js (Markdown parser), Google Guice  
**Storage**: PostgreSQL (production), SQLite (testing/local via H2); encrypted user data at rest using centralized encryption key  
**Testing**: JUnit 5 for backend, MockK for mocking; Jest/React Testing Library for frontend  
**Target Platform**: Web application (single-page application with server-side rendering support)  
**Project Type**: Web service (Ktor backend + bundled React frontend)  
**Performance Goals**:
- Note creation within 2 seconds of page load
- Search results in <500ms for 1000 notes
- API responses in <300ms for list operations
- Support 10,000+ notes per user without UI degradation  
**Constraints**:
- All user data encrypted at rest using centralized server encryption key (configurable via environment variable)
- File uploads configurable size limit via environment variable
- Graceful degradation when AI service unavailable
- Auto-save enabled on all note edits  
**Scale/Scope**: Single-user focused initially; multi-user support via Google OAuth; unlimited notes per user

## Build System Configuration

### React Frontend Integration

The React frontend will be built as a separate project within the repository and bundled with Ktor at build time.

**Directory Structure**:
```
frontend/
├── package.json
├── vite.config.ts
├── src/
│   ├── main.tsx
│   └── components/
└── build/          # Output directory (generated by npm run build)
```

### Build Process

1. **React Compilation**: `npm run build` in the `frontend/` directory generates static assets in `frontend/build/`
2. **Ktor Bundling**: The `buildFatJar` task copies React build output to `src/main/resources/static/`
3. **Final Artifact**: Single JAR containing both Kotlin backend and React frontend

### Implementation Plan for Gradle Build

**Approach**: Use the Gradle Node plugin to run npm commands during the build process.

```kotlin
// Add to build.gradle.kts (to be implemented)
plugins {
    id("com.github.node-gradle.node") version "7.0.1" apply false
}

tasks.register<Exec>("compileReact") {
    workingDir = File(project.projectDir, "frontend")
    commandLine("npm", "run", "build")
}

tasks.named<Jar>("jar") {
    dependsOn("compileReact")
    from(File(project.projectDir, "frontend/build")) {
        into("static")
    }
}
```

**Key Considerations**:
- React build runs before JAR creation
- Output directory `frontend/build/` is copied to Ktor's static resources path
- Build only proceeds if frontend dependencies are installed (npm install on first run)

## Phase 1: Design & Contracts

### Data Model

**Entities to be defined**:
- `User`: OAuth identity, API key, encryption salt
- `Note`: Title, content (Markdown), tags, status (draft/published), version history
- `Tag`: Name, user reference, many-to-many with notes
- `Attachment`: File metadata, encrypted storage path, note reference
- `APIToken`: Per-user API credentials
- `Version`: Note history tracking

**State Transitions**:
- Draft → Published (auto-save on content change)
- Pending AI Processing → Processed (async job completion)

### Contracts

**REST API Endpoints**:
- `POST /api/v1/notes` - Create note
- `GET /api/v1/notes` - List notes with pagination
- `GET /api/v1/notes/{id}` - Get note by ID
- `PUT /api/v1/notes/{id}` - Update note
- `DELETE /api/v1/notes/{id}` - Delete note
- `POST /api/v1/notes/{id}/attachments` - Upload attachment
- `GET /api/v1/notes/{id}/attachments/{attachmentId}` - Download attachment
- `POST /api/v1/search` - Search notes
- `GET /api/v1/tags` - List user tags

**WebSocket Events** (if real-time needed):
- Note updates broadcast to owner

### Quickstart Guide

**For Developers**:
1. Clone repository
2. Run `./gradlew run` for development server
3. Access at http://localhost:8080
4. For testing with embedded database, no additional setup required
